# Design Context

This file is the canonical reference for design decisions across all
maximal UI surfaces (the Tauri windows and the proxy-served HTML
pages). It is consulted by the design-* skills (design-frontend,
design-onboard, design-critique, design-check, etc.) and by any agent
or human building UI for this project.

## Users

maximal is a **local proxy** between developer tooling (Claude Code,
the Anthropic SDK, the OpenAI SDK, opencode, custom scripts) and the
GitHub Copilot API. Two access modes converge on the same files and
behavior:

- **CLI users** — comfortable in the terminal, prefer keyboard,
  expect machine-readable output and explicit config. `maximal auth`,
  `maximal start`, `maximal debug --json` are their primary surface.
- **Menu-bar app users** — same audience, different moment. Run
  maximal as a set-and-forget background utility from `/Applications`,
  with a tray icon and occasional "what's it doing" peeks. The GUI
  exists for the moments where typing is the wrong tool: first-run
  auth (no terminal), occasional usage check, settings tweak.

Both paths read and write the same `~/.local/share/maximal/`
directory. Neither audience needs the other's affordances; both
should feel like the tool was built for them.

## Brand Personality

**Warm + crafted + considered.** The brand mark (red rounded square
in `#c8334a` with a wonky Fraunces "m") sets the tone: a tool made
by a person, not a corporation. Friendly without being playful-cute.
Confident without being terminal-stark.

- Voice: clear, direct, second-person. "Sign in with GitHub" not
  "Initiate OAuth flow." "We can't reach the proxy" not "ECONNREFUSED."
- Type: a sans-serif body (system font stack) does the heavy lifting;
  Fraunces appears only in the brand mark and possibly one accent
  heading per window.
- Restraint over decoration: layout, type, and spacing carry the
  feeling. Color is one surface, not the only surface.

## Aesthetic Direction

**Humanist powerful.** Dense with capability but never overwhelming.
Doesn't push machine concerns onto humans (no raw config keys in the
primary UI, no JSON dumps). Doesn't push human concerns into machine
shapes (no chat UI for what should be a single button).

### Reference apps (starting points, not destinations)

- **Ollama Settings**: card-grouped sections, monochrome utility
  icons paired with a single humanist accent (the drawn llama avatar).
- **Claude / Anthropic Settings**: sidebar nav with restrained active
  state, human-tone copy ("What should Claude call you?"), monitor /
  sun / moon theme toggle.
- **Anti-references**: Linear (too cold and dense), Slack (too noisy),
  the `create-tauri-app` starter (too generic).

### Layout pattern

Hybrid, scaled by window breadth:

- **Settings window** (8 sections) — left-rail nav + right-pane
  content. Sections inside the pane are separated by **typography
  and spacing**, not by card chrome. Cards are reserved for
  per-entity rows where they apply (e.g., one card per provider in
  the Providers section, one card per API key in the API clients
  section). See "When to use a card" below.
- **Setup and Dashboard windows** — single-section, scroll-only, no
  sidebar. Same rule: typographic sectioning between concerns; cards
  only when a card represents one discrete actionable entity.

### Theme

- **Both light and dark**, with explicit override in Settings (matches
  Anthropic's monitor / sun / moon toggle). System (`prefers-color-scheme`)
  is the third option and the default.
- **User-themable accent + surface colors.** The brand red is the
  default identity color, but users can dial in their own UI color
  and icon color in Settings.
- **Contrast is ours, not theirs.** When a user picks a combination
  whose body-text contrast drops below WCAG AA (4.5:1), surface a
  warning chip near the affected control. Never block — the user is
  in charge — but be honest about the consequence.

### Density

Comfortable. Slightly more spacious than Raycast, far less airy than
Bear. Rows have real padding, line-height ≥ 1.4 for body, section
gaps generous enough that the eye breathes. Power is in depth (more
sections, collapsible advanced) rather than density (more items per
row).

### When to use a card

A card is a container with chrome (background fill, border or shadow,
internal padding, rounded corners). Cards exist to make a **discrete
actionable entity** visually cohesive — one card represents one
*thing the user can act on as a unit*.

**Use a card when:**

- The contents are the state of a single entity (a provider, an API
  key row, a connected account). The card's borders say "this group
  of fields belongs together; act on them together."
- The user can perform actions specific to that entity (toggle on/off,
  edit, delete) without affecting neighbors.

**Do not use a card for:**

- **Sectioning content within a window.** That's typography's job —
  a strong section heading, generous vertical space, an optional thin
  rule. Wrapping every section in a card adds chrome without
  meaning, dilutes the strong-grouping signal when cards *are*
  warranted, and reads as the AI-dashboard archetype (per the
  design-frontend AI-slop screen).
- **Holding a single block of content** (a chart, a code sample, an
  empty state). The content's own visual already separates it.
- **Stacking icons + heading + text in identical rows.** That's a
  list, not a grid of cards.

**Card nesting is forbidden.** If you find yourself wanting cards
inside cards, the inner card should be a list-row inside the outer
card, or the outer card should be a typographic section. Either
collapses the nesting; pick one.

**Squint test:** if you blur your eyes and the page looks like a
grid of similar rectangles, the cards aren't doing work — they're
chrome. Drop them.

### Iconography

- **Functional / utility icons**: monochrome, system-tinted, stroke
  ~1.5–2px at 16/20/24 sizes. Lucide or Phosphor as the source
  library; pick one and stick with it.
- **Identity / accent**: the brand "m" appears once per window (top
  of the window, near the heading). Never on every row.

## Design Principles

These five principles override all other guidance when they conflict.

### 1. Speak to the person, not the file

Settings labels and inline copy describe what the control does in
terms a person would use, not the field name in `config.json`.
"Sign in with GitHub" beats "Authenticate via OAuth device-code flow."
"Auto-download updates" beats "autoDownloadUpdates: bool."

### 2. Power lives in depth, not density

Common controls are big, clear, and one click away. Advanced
controls live one level down (collapsed sections, "Advanced" tab,
"Reveal in editor" escape hatch). Don't put everything on screen
because we can.

### 3. Color is the user's, contrast is ours

Honor user-chosen colors for surface and icon. Always compute and
surface contrast for text and controls; warn (don't block) when a
combination drops below WCAG AA. The brand mark stays the brand red
in the dock and tray — that's identity, not preference.

### 4. One humanist accent per window

The brand "m" or a hand-drawn touch appears once per window — to
remind the user it was made by a person, not on every row. Excess
humanism is noise; rationed humanism is character.

### 5. Reduced motion is a contract, not a hint

When `prefers-reduced-motion: reduce` is set, animation drops to
opacity crossfades or instant transitions. Hover scales, slide-ins,
spring physics — all off. The setting is honored literally, not
"with reduced intensity."

## Type system

**Pairing: E1 — Fraunces (display) + Commissioner (body).** Humanist
editorial pair from the approved table. Fraunces is already in the
brand mark, so the display tier and the icon share a typographic
family. Commissioner is the workhorse humanist sans for everything
else: warm letterforms, broad weight range (100–900), designed for
app UI legibility at small sizes.

**Ramp: Standard App** (16px base, 1.2 ratio). The full token block
below is the source of truth — never inline raw pixel/rem values in
components; reference these tokens.

```css
:root {
  /* ---- Font stacks ---- */
  --font-display: "Fraunces", Georgia, "Times New Roman", serif;
  --font-body:    "Commissioner", "Segoe UI", Helvetica, Arial, sans-serif;
  --font-mono:    ui-monospace, SFMono-Regular, "SF Mono", Menlo, Consolas, "Liberation Mono", monospace;

  /* ---- Standard App Type Ramp (base 16px, ratio 1.2) ---- */
  --text-xs:   0.75rem;    /* 12px — caption, footnote, helper text */
  --text-sm:   0.875rem;   /* 14px — inline labels, dense rows */
  --text-base: 1rem;       /* 16px — body, inputs, control labels */
  --text-md:   1.125rem;   /* 18px — lead body, one-section windows */
  --text-lg:   1.25rem;    /* 20px — subhead, sidebar-nav active item */
  --text-xl:   1.5rem;     /* 24px — section heading inside a window */
  --text-2xl:  2rem;       /* 32px — window heading (H1) */
  --text-3xl:  2.5rem;     /* 40px — display, reserved for onboarding moments */
  --text-4xl:  3rem;       /* 48px — hero display (rare; see Welcome state in Setup PRD) */

  --weight-base: 400;
  --weight-md:   500;
  --weight-lg:   600;
  --weight-xl:   600;
  --weight-2xl:  700;

  --leading-xl:   1.3;

  --leading-4xl:  1.1;

  --tracking-base: 0em;
  --tracking-lg:  -0.005em;

  font-family: var(--font-body);
```

### Font loading

Both Fraunces and Commissioner are on Google Fonts and need to be
loaded explicitly. Inline the `@font-face` declarations from the
self-hosted files (preferred — no third-party request from the
Tauri webview) or, during prototyping only, link the Google Fonts
stylesheet with `display=swap`:

```html
<link rel="preload" as="style" href="..."/>
<link rel="stylesheet"
      href="https://fonts.googleapis.com/css2?family=Fraunces:opsz,wght,SOFT,WONK@9..144,300..900,30,1&family=Commissioner:wght@200..900&display=swap"/>
```

**For production**: self-host. The Tauri shell ships the WOFF2 files
bundled with the Vite output so the webview never makes an external
request — keeps the app working offline and preserves the
no-telemetry posture.

### Weights

- **400** — body default
- **500** — emphasis, button label, active sidebar-nav item
- **600** — section headings (`h2`, `.h-section`)
- **700** — window headings (`h1`, `.h-display`)
- **800** — reserved for the hero display tier only

No weight 100/200/300 — reads thin on dark backgrounds. No 900 —
too heavy for warm+crafted. The bottom of Commissioner's range
(200) is technically available but unused.

### Mono usage

`var(--font-mono)` is for: code samples (the `curl` block in the
Dashboard's Connect section), inline API keys, the device code in
the Setup window's Waiting state, and the numeric tails in the
Activity feed. Always `tabular-nums` when paired with updating
values so columns don't dance.

### Lengths and density

- Body text: minimum size `--text-base` (16px); never smaller for
  multi-line prose.
- Max line length: `65ch` on prose containers. Code/sample blocks
  are exempt and allowed to scroll.
- Setup and Dashboard windows: prose uses `--text-md` (18px) because
  the column is narrower and the spaces breathe more.
- Settings: dense form rows use `--text-base` for control labels and
  values; `--text-sm` for descriptions.

### What changed from the previous version

The earlier draft used the browser default stack as the primary body
font — explicitly disallowed by the project's design-frontend rules
because it renders generic and fingerprints AI-built work. Replaced
entirely with the E1 pairing. Tokens renamed from `xs/sm/base/md/...`
in the previous prose table to the CSS-custom-property form
(`--text-xs`, `--text-sm`, …) so the ramp is consumed by reference,
not by retyping the same numbers per component.

### Numerics

- **Tabular figures** (`font-variant-numeric: tabular-nums`) for any
  numbers that update in place: rate-limit counters, request durations,
  token counts, the activity feed timestamps.
- **Currency / measurement units** sit immediately adjacent (no extra
  space, `7,432 tokens`, `1.2s`, not `7,432 tokens` with a wide gap).

### Casing

- Sentence case for everything user-facing: headings, buttons, menu
  items, labels. No ALL CAPS labels (too marketing).
- Title Case only in proper nouns ("GitHub Copilot", "Open Maximal").
- Section dividers like "Account" / "API clients" — sentence case
  for the second word.

### Emphasis

- **Bold** (weight 500–600) for in-copy emphasis. Sparingly.
- Italics for the rare phrase or quoted user input. Avoid for UI labels.
- Never combine bold + italics. Choose one.

## Layout system

### Window sizes

| Window     | Default | Min      | Max      | Resizable | Notes |
|------------|---------|----------|----------|-----------|-------|
| Setup      | 520×620 | 480×560  | 720×800  | yes       | Single column, vertical scroll on overflow |
| Dashboard  | 960×720 | 720×560  | 1400×1000| yes       | One scrollable column with sectioned content |
| Settings   | 880×720 | 720×560  | 1200×900 | yes       | Sidebar (200px) + content pane |

Single-instance for all three (re-show + focus the existing window
instead of opening another). Position: center on first launch, then
respect last position.

### Grid + columns

Desktop only — no responsive breakpoints in v1. If a user makes the
window narrower than the min, the OS prevents it; we don't have to
build a mobile fallback.

- **Setup window**: single column, content max 440px wide, centered.
- **Dashboard window**: single column, content max 720px wide, left-
  aligned with 24px gutters either side; cards span the full content
  width.
- **Settings window**: 200px sidebar + content pane. Content pane max
  640px, left-aligned. Within the pane, form rows use a 33% label /
  66% control split.

### Spacing scale

```css
--space-1: 4px;
--space-2: 8px;
--space-3: 12px;
--space-4: 16px;
--space-5: 24px;
--space-6: 32px;
--space-7: 48px;
--space-8: 64px;  /* reserved for inter-section gaps on large windows */
```

Stick to these. Inline gaps inside a row: 8/12. Card-internal
padding: 16/20. Card-to-card gap: 16. Section gap inside a window:
24/32. Between window edge and content: 24 (Dashboard, Setup) or 32
(Settings content pane).

### Surfaces (3 levels)

```
                    Dark           Light
--surface-base      #0a0a0a        #fafafa     window background
--surface-card      #161616        #ffffff     cards, sidebar
--surface-control   #1f1f1f        #f0f0f0     inputs, buttons
```

These are defaults; the user-themable accent overrides
`--surface-card` and `--surface-control` deltas independently if they
diverge from the brand. The system computes contrast against
`--surface-card` (where most text sits) and warns when text-on-card
drops below WCAG AA.

### Elevation (3 levels)

```css
--elevation-card:   0 1px 2px rgb(0 0 0 / 0.06);
--elevation-modal:  0 8px 24px rgb(0 0 0 / 0.18);
--elevation-tooltip:0 2px 6px rgb(0 0 0 / 0.10);
```

In dark mode the shadows are nearly invisible; the surface step
itself does the lifting. Don't add a fourth level.

### Radii

```css
--radius-input: 6px;
--radius-card:  8px;
--radius-chip:  4px;
--radius-pill:  9999px;  /* reserved for status dots / count badges */
```

### Component dimensions

**Buttons**

| Variant | Height | Px padding | Font | Weight | Notes |
|---------|--------|-----------|------|--------|-------|
| Primary | 36     | 16        | 14   | 500    | White on accent (light) / accent on dark; full opacity |
| Secondary | 36   | 16        | 14   | 500    | Outline 1px, transparent fill |
| Ghost   | 32     | 12        | 13   | 500    | No outline; hover surface |
| Icon    | 32×32  | —         | —    | —      | 18px icon centered |
| Small   | 28     | 12        | 13   | 500    | Sparingly; only in dense rows |

**Inputs**

| Variant   | Height  | Padding | Font | Notes |
|-----------|---------|---------|------|-------|
| Text      | 36      | 12      | 14   | 1px border, focus ring 2px offset |
| Textarea  | min 96  | 12      | 14   | Resize vertical only |
| Select    | 36      | 12 / 32 | 14   | Right-side chevron 16px |
| Switch    | 24×16   | —       | —    | iOS-style; 200ms ease |
| Checkbox  | 16×16   | —       | —    | 2px stroke |

**Form rows**

- Min height: 48px (single-line label + control).
- Label column: ~210px on Settings, ~33% of content elsewhere.
- Control column: remainder; controls right-aligned only when they're
  visually right-anchored (toggles, dropdowns); left-aligned when
  they're text inputs.

**Sidebar (Settings)**

- Width: 200px fixed.
- Padding: 8px vertical, 4px horizontal.
- Nav item: 36px tall, 8px vertical padding, 12px horizontal, weight
  400 → 500 on active. Active state: a 1px-rounded surface step (not
  a left bar; that reads more dev-tool than humanist).
- Group label (e.g. "Desktop app" in the Anthropic ref): 12px, 500,
  letter-spacing 0.02em, uppercase, color = muted text. One per
  group, not on every item.

**Cards**

- Padding: 16 (vertical) × 20 (horizontal). One step tighter when
  nested inside a card.
- Gap between cards in a stack: 16.
- Internal row gap: 12 (compact rows like a toggle list) or 16
  (form rows with their own labels).

### Z-axis order

```
0    base / card content
10   sticky window header (Settings sidebar's top, Dashboard status strip)
100  dropdowns, popovers, autocomplete results
200  toasts ("Saved", "Copied to clipboard")
300  modals (confirmation dialogs)
400  tooltips (always on top so they're never occluded)
```

### Focus rings

- Always visible on keyboard navigation; hidden on mouse use via
  `:focus-visible`.
- 2px offset, 2px ring width, color = accent (with contrast fallback
  to `--text-strong` when accent on the current surface drops below
  3:1).
- Identical ring on all focusable elements — buttons, inputs,
  switches, links, nav items. No special-case ring styles.

### Motion

- Transitions default to 150ms ease-out for hover/active state changes,
  200ms for layout shifts (window resize, card collapse).
- `prefers-reduced-motion: reduce` → all timings drop to 0 except a
  60ms opacity crossfade where a transition would otherwise be
  jarring. See principle #5.
- No bounce / spring / elastic curves. No parallax. Motion is utility,
  not delight — delight comes from typography, spacing, and copy.

## Operational notes

- WCAG AA target across all colors, contrast, and focus rings.
- Default theme keys: `--text-strong`, `--text-base`, `--text-muted`,
  `--accent`, `--accent-fg`, `--surface-{base,card,control}`,
  `--border-subtle`, `--border-strong`. User overrides apply to
  `--accent` and `--surface-card`; everything else recomputes against
  the new pair with the contrast guardrails described above.
- All numeric tokens (spacing, type sizes, radii, elevations) ship as
  CSS custom properties on `:root`, with `[data-theme="light"]` /
  `[data-theme="dark"]` overrides on the surface and text keys only.
- One token table, one place to look. Don't redeclare values inline
  in components; if you need a new value, add a token first and
  justify it in this file.
