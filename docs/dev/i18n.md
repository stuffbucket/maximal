# i18n — UI strings & OS-conditional terminology

A client-side i18n layer for the settings webview. Its first job was a bug:
the shell hardcoded "Finder", so the Windows parity build shipped "Reveal
logs in Finder" on machines that have File Explorer. Rather than scatter
`if (isWindows)` string branches, OS-conditional terms live in one catalog and
render through ICU MessageFormat — the same format Apple String Catalogs and
.NET localize with, so the OS fix is also the first slice of a real
multi-language catalog rather than a throwaway.

## Where strings live

`shell/src/i18n/en.json` — one catalog, ICU MessageFormat values:

```json
{
  "file-manager": "{os, select, windows {File Explorer} linux {Files} other {Finder}}",
  "app-container": "{os, select, macos {menu bar} other {system tray}}",
  "reveal-logs": "Reveal logs in {fileManager}",
  "reveal-config": "Reveal config in {fileManager}…"
}
```

It is the **single source of truth** for OS-conditional terminology across the
whole product (JS UI and the Rust tray/notification side both defer to it).

### Why ICU

- **Platform parity.** ICU MessageFormat is what Apple String Catalogs and .NET
  localization compute on. Modeling terms the way first-party platforms do
  keeps us aligned with the tooling translators already use.
- **Forward-compat.** `select`/`plural` handle the grammar (gender, count, word
  order) that other languages need. Building the OS fix on ICU means adding a
  language later is additive, not a rewrite.

## The two OS-conditional terms and how `{os}` is sourced

| Key | Renders as |
|---|---|
| `file-manager` | Finder / File Explorer / Files |
| `app-container` | menu bar (macOS) / system tray (Windows, Linux) |

Both are ICU `select` on an `{os}` argument. The argument comes from
`osKind()` in `shell/src/ui/platform.ts`, which normalizes the OS to
`macos | windows | linux`.

`osKind()` reads the **user-agent** (preferring `navigator.userAgentData.platform`,
falling back to the UA string) rather than a Tauri OS plugin. This is
deliberate: the embedded UI is the same bundle on every platform, and it can be
opened both in the Tauri webview **and** a plain browser. A dependency-free
UA signal works in both; a Tauri-only plugin would not.

`t()` and `data-i18n` inject `{os}` (and the resolved `{fileManager}` noun)
automatically, so call sites never wire them up by hand.

## How to add a UI string

1. Add a key to `shell/src/i18n/en.json` with an ICU MessageFormat value.
2. Render it, either:
   - **In code:** `import { t } from "./i18n"` then `t("my-key")`, or
     `t("my-key", { count: 3 })` to pass placeholders. `{os}` and
     `{fileManager}` are injected for you.
   - **In markup:** add `data-i18n="my-key"` to any element — `applyI18n()`
     (in `shell/src/main.ts`) fills its text content at boot, before first
     paint. No extra wiring per element.

An unknown key renders as the key itself (fails visibly, never throws over a
UI string).

## How to add a language later

The layer is built to make this additive — one seam, no call-site changes:

1. Drop in `shell/src/i18n/<locale>.json` with the **same keys** as `en.json`.
2. Register it in the `CATALOGS` map in `shell/src/i18n/index.ts`.
3. Make `resolveLocale()` return the desired locale (e.g. from
   `navigator.language` or the proxy-reported locale). It is hardcoded to
   `"en"` today and is the **only** thing multi-language work swaps.

See the deferred follow-up issue for the full multi-language plan (locale
selector, translation workflow).

## The Rust mirror

The Tauri tray/notification side (`shell/src-tauri/src/lib.rs`) needs the
`app-container` noun too, but pulling an ICU runtime into Rust for two words is
not worth it. Instead `app_container_noun()` is a trivial `cfg!(target_os)`
hand mirror of the catalog term:

```rust
fn app_container_noun() -> &'static str {
    if cfg!(target_os = "macos") { "menu bar" } else { "system tray" }
}
```

A hand mirror can silently drift from the catalog, so
`tests/i18n-catalog-parity.test.ts` pins both directions: the catalog still
spells the nouns as the code expects, **and** the Rust mirror still carries
those exact literals. If either drifts, the test fails and points back at the
catalog as the source of truth.

## The one hard rule

**Never concatenate a term into a sentence — pass it as an ICU argument.**

```
✅  "reveal-logs": "Reveal logs in {fileManager}"
❌  "Reveal logs in " + fileManagerName()
```

Concatenation freezes English word order and breaks the moment a language
orders the sentence differently. Passing the term as an argument keeps the
whole phrase translatable. This is the invariant that stops the OS fix from
becoming a corner we have to unwind later.
