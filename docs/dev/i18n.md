# i18n — UI strings & OS-conditional terminology

A client-side i18n layer for the embedded webviews (settings **and** the usage
dashboard). It started as a bug fix — the shell hardcoded "Finder", so the
Windows parity build shipped "Reveal logs in Finder" on machines that have File
Explorer — and grew into a full multi-language catalog. Rather than scatter
`if (isWindows)` string branches, every user-facing string lives in one catalog
and renders through ICU MessageFormat — the same format Apple String Catalogs
and .NET localize with — so OS-conditional wording and real translation are the
same mechanism, not two.

## Where strings live

`shell/src/i18n/en.json` is the **base catalog** and single source of truth — a
flat map of kebab-case keys to ICU MessageFormat values, grouped by section
prefix (`nav-`, `account-`, `endpoint-`, `logs-`, `diagnostics-`, `uninstall-`,
`dashboard-`, `common-`, …). Every user-facing string in both webviews has a key
here; the Rust tray/notification side defers to it too.

```json
{
  "file-manager": "{os, select, windows {File Explorer} linux {Files} other {Finder}}",
  "app-container": "{os, select, macos {menu bar} other {system tray}}",
  "reveal-logs": "Reveal logs in {fileManager}",
  "logs-retention-value": "{n, plural, one {# day} other {# days}}, then deleted on next boot"
}
```

### Why ICU

- **Platform parity.** ICU MessageFormat is what Apple String Catalogs and .NET
  localization compute on. Modeling terms the way first-party platforms do keeps
  us aligned with the tooling translators already use.
- **Grammar, not concatenation.** `select`/`plural` handle count, gender, and
  word order that other languages need — so a translation can reword freely
  around a `{placeholder}` instead of being frozen to English order.

The JS side renders via `intl-messageformat` (a shell dependency that leans on
the webview's built-in `Intl`). The runtime lives in `shell/src/i18n/index.ts`.

## Locales we ship & the fallback chain

Twelve catalogs are registered in `CATALOGS` (`shell/src/i18n/index.ts`):

| File | Role |
|---|---|
| `en.json` | base — US English, the full key set (source of truth) |
| `en-GB.json` | sparse overrides (spelling: licence, authorisation, catalogue) |
| `es.json` | **full** Spanish language base (neutral, formal register) |
| `es-MX.json` | sparse Mexican-Spanish overrides (empty today — `es` covers it) |
| `es-ES.json` | sparse Peninsular-Spanish overrides |
| `zh.json` | **full** Simplified Chinese base |
| `fr.json` | **full** French base |
| `de.json` | **full** German base (formal *Sie* register) |
| `ru.json` | **full** Russian base (`one/few/many/other` plurals) |
| `ja.json` | **full** Japanese base |
| `it.json` | **full** Italian base |
| `pt.json` | **full** Portuguese base (BR-leaning neutral) |

Every non-English base is a full 286-key catalog. Lookup is **resource
fallback**: a resolved locale resolves each key through
`region → language → en`. e.g. `es-MX` tries `es-MX` → `es` → `en`. That's why
regional files are sparse (or empty) — omit a key and it falls back. `es` is a
*full* base precisely so `es-MX`/`es-ES` users never fall all the way to
English. A key missing everywhere renders as the key itself (fails visibly,
never throws over a UI string).

## How the active locale is chosen

`resolveLocale()` picks, in order:

1. A persisted user override — `localStorage["maximal.locale"]`, set by the
   locale picker (the `<select data-locale-select>` in each webview header,
   wired by `wireLocalePicker()`), if it's a shipped tag.
2. A best-fit match of `navigator.languages` (full BCP-47 tag, then
   language-only) against the shipped locales.
3. `"en"`.

The override is shared across both webviews (same `:4141` origin), so switching
language on one surface shows on the other's next load. Switching via the picker
re-runs `applyI18n(document)` and the surface's repaint, so it updates live.

`{os}` (and the resolved `{fileManager}` noun) are injected automatically by
`t()` / `applyI18n`, sourced from `osKind()` in `shell/src/ui/platform.ts` — the
**user-agent** (not a Tauri OS plugin), because the same bundle runs in both the
Tauri webview and a plain browser, so a dependency-free UA signal is the only
one that works in both.

## How to add a UI string

1. Add a key to `shell/src/i18n/en.json` with an ICU MessageFormat value.
2. Render it:
   - **In markup:** `data-i18n="my-key"` on any element fills its text;
     `data-i18n-aria-label` / `-title` / `-placeholder` fill those attributes;
     `data-i18n-count="7"` on the same element feeds an ICU `{n}` plural. The
     shared `applyI18n()` (`shell/src/i18n/apply.ts`, used by both entries)
     sweeps them before first paint.
   - **In code:** `import { t } from "./i18n"` then `t("my-key")` or
     `t("my-key", { count: 3 })`. `{os}`/`{fileManager}` are injected for you.
3. Translate: add the key to `es.json` (the full base). Add `en-GB`/`es-MX`/
   `es-ES` overrides only where the wording actually differs — otherwise the
   fallback chain covers it.

## How to add another language

Additive — one seam, no call-site changes:

1. Drop in `shell/src/i18n/<locale>.json` (a full language base like `es`, or a
   sparse regional override of an existing base).
2. Register it in `CATALOGS` and give it a label in `localeLabel()`
   (`shell/src/i18n/index.ts`). It then appears in every picker automatically.

## The Rust (native) side

The Tauri shell's OS-drawn chrome — tray menu, native notifications, native
dialogs, and the Settings/Dashboard window titles — is localized too, but
without an ICU runtime. `shell/src-tauri/src/native_i18n.rs` `include_str!`s the
SAME `shell/src/i18n/*.json` catalogs and reads the `native-*` keys out of them,
with plain `{name}` placeholder substitution (no `plural`/`select`). Any
OS-conditional noun is baked into a per-OS catalog key (e.g.
`native-notify-startup-body-macos` vs `-other`), so the caller just picks the
key by `cfg!(target_os)` — no runtime `select` needed.

### Native strings follow the locale picker

The webview persists its choice to `localStorage["maximal.locale"]` and, on
boot and on every picker change, pushes it across the IPC boundary via the
`set_locale` command. That updates the shell's `LocaleState` and rebuilds the
tray so tray/tooltip re-resolve immediately; later notifications, dialogs, and
window titles read the same state. Before any webview exists (e.g. the
first-launch "Maximal is running" banner) the shell falls back to the OS locale
via `native_i18n::resolve_locale` seeded at setup.

## What the parity test guards

`tests/i18n-catalog-parity.test.ts` is dependency-free and pins:

1. **OS terms** — the catalog still spells `app-container` / `file-manager` as
   the JS ICU call sites expect.
2. **Multi-locale integrity** — for every locale file: no key absent from the
   `en` base (no dead/typo'd keys), balanced ICU braces, and the same
   placeholder / plural-select argument surface as its base key (dropping an arg
   loses data; introducing one throws at format time). `es` must additionally
   cover **every** base key (it's the language base others fall back to).

## The one hard rule

**Never concatenate a term into a sentence — pass it as an ICU argument.**

```
✅  "reveal-logs": "Reveal logs in {fileManager}"
❌  "Reveal logs in " + fileManagerName()
```

Concatenation freezes English word order and breaks the moment a language orders
the sentence differently. Passing the term as an argument keeps the whole phrase
translatable — the invariant the whole layer rests on.
