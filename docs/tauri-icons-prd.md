# PRD: Replace default Tauri icons with the maximal brand

## Problem

`shell/src-tauri/icons/` currently contains the **stock `create-tauri-app` starter icons** â€” the cute Tauri "T" logo, regenerated by every prior `bunx tauri icon` invocation with no custom input. Two visible consequences:

1. **Dock / app switcher / Spotlight** show the Tauri logo, not maximal.
2. **macOS menu-bar tray** uses `app.default_window_icon()` (i.e. the full-color bundle icon) as the tray icon â€” full-color, no template-rendering, no dark-mode flip. Looks foreign on the menu bar.

We already have a canonical brand asset in the repo: `build/macos/app-icon.svg` â€” a 1024Ã—1024 rounded square in `#c8334a` with the word "maximal", currently rendered to `build/macos/app-template/Contents/Resources/AppIcon.icns` for the CLI `.app` bundle. The Tauri shell should reuse it.

## Goals

- Replace every icon under `shell/src-tauri/icons/` with renders generated from the existing brand SVG.
- Add a dedicated **template tray icon** so the menu bar looks native on macOS (monochrome silhouette, tinted by the system, dark-mode aware).
- Document the regeneration command so future brand updates are a one-liner.
- No new graphics work in v1 â€” reuse what we have.

## Non-Goals

- Redesigning the brand. The red rounded square stays.
- Refreshing the site favicon (`site/public/favicon.svg`) or the CLI's bundle icon. Both already point at the right thing.
- Windows installer icon (`build/windows/maximal.wxs`). Separate codepath, separate PR if it drifts.
- Animated / state-indicator tray icons (badged for "needs setup," dot for "running," etc.). The first-run-setup PRD mentions a state badge â€” that's a v2 concern after the static icon lands.

## Source of Truth

**One canonical 1024Ã—1024 source: `build/macos/app-icon.svg`.**

Why this one over the site favicon:
- Already wired into the CLI's `.app` build, so the desktop install and the Tauri install render identically.
- 1024Ã—1024 viewBox â€” exactly the size `tauri icon` wants. The site favicon is `32Ã—32` viewBox, meant for browser tabs.
- Brand consistency: dock + menu bar + DMG installer chrome all match.

The tray template icon is a **separate** asset because macOS menu-bar icons should be flat monochrome with alpha (the OS tints them). We'll derive a stripped-down "M" silhouette from the same brand, committed as `shell/src-tauri/icons/tray.svg`.

## Generation

### Bundle icons (the 17 files)

`tauri icon` ships in the Tauri CLI and renders the full sizes Tauri's bundler expects from a single source.

```sh
# From shell/
bunx tauri icon ../../build/macos/app-icon.svg \
  --output src-tauri/icons
```

This regenerates (overwrites in place):
- `32x32.png`, `128x128.png`, `128x128@2x.png`, `icon.png` (512Ã—512)
- `icon.icns` (macOS, multi-resolution)
- `icon.ico` (Windows, multi-resolution)
- Microsoft Store `Square{30,44,71,89,107,142,150,284,310}Logo.png` and `StoreLogo.png` (`50Ã—50`)

Add a wrapper npm script in the root `package.json`:

```json
"app:icons": "cd shell && bunx tauri icon ../build/macos/app-icon.svg --output src-tauri/icons"
```

So future brand changes are: edit `build/macos/app-icon.svg` â†’ `bun run app:icons` â†’ commit.

### Tray template icon

`tauri icon` doesn't produce a template variant. We commit the source SVG and render two PNGs by hand (or via a tiny script in `scripts/render-tray-icon.ts` using `sharp` or `resvg-js` if we want to avoid a checked-in binary):

- `shell/src-tauri/icons/tray.png` (`22Ã—22`)
- `shell/src-tauri/icons/tray@2x.png` (`44Ã—44`)

Both monochrome (#000) with full alpha. macOS applies its own tint at runtime when `icon_as_template(true)` is set.

Initial v1: a stylized "M" matching the brand's strokes, with `~2px` outer padding to match the optical sizing of native menu-bar icons (Spotlight, Wi-Fi, etc., are ~17Ã—17 ink in a 22Ã—22 box).

If hand-rolling a tray SVG is contentious, a fast fallback is a black silhouette of the existing app-icon glyph, alpha 100%, white background removed.

## Wiring

### tauri.conf.json â€” no change

The `bundle.icon` array already references the right filenames:

```json
"icon": [
  "icons/32x32.png",
  "icons/128x128.png",
  "icons/128x128@2x.png",
  "icons/icon.icns",
  "icons/icon.ico"
]
```

Regeneration overwrites these in place. Bundler picks them up automatically.

### shell/src-tauri/src/lib.rs â€” tray icon

Today (line ~159):

```rust
.icon(app.default_window_icon().unwrap().clone())
```

Replace with the dedicated template icon, scoped to macOS (other platforms keep using the bundle icon, which is the right default there):

```rust
use tauri::image::Image;

let tray_icon = Image::from_path(
    app.path()
        .resolve("icons/tray.png", tauri::path::BaseDirectory::Resource)?,
)?;

let mut builder = TrayIconBuilder::with_id(TRAY_ID)
    .menu(&menu)
    .show_menu_on_left_click(true)
    .tooltip("maximal")
    .icon(tray_icon);

#[cfg(target_os = "macos")]
{
    builder = builder.icon_as_template(true);
}

builder.build(app)?;
```

The 2x PNG isn't loaded explicitly â€” `Image::from_path` uses Tauri's resource layer, which understands `@2x` adjacency on macOS Retina.

Resource bundling: add `"icons/tray.png"` and `"icons/tray@2x.png"` to `tauri.conf.json`'s `bundle.resources` if they aren't already covered by the existing `bundle.icon` discovery. (Verify against Tauri 2's actual behavior â€” recent versions auto-include the `icons/` folder.)

## Validation

After regeneration:

1. **`bun run app:build`** completes without warnings about missing icon files.
2. **Open `shell/src-tauri/target/release/bundle/macos/maximal.app`** â€” Finder shows the red rounded-square brand. Dock/app-switcher likewise.
3. **DMG render** â€” `bundle/dmg/maximal_*.dmg` shows the brand icon under the drag-to-Applications graphic.
4. **Tray (light mode + dark mode)** â€” monochrome "M" silhouette, system-tinted; readable in both. Macbook with default settings; verify against a system with menu bar set to "Dark."
5. **`tauri dev`** â€” same visual result as the built `.app`. Confirms we're not in a "build-only icon" trap.
6. **Spotlight ("Maximal")** â€” first hit shows the brand icon, not the Tauri logo.

## Risks

- **`tauri icon` produces oversaturated PNGs at small sizes** if the source SVG has thin strokes. The existing brand is a chunky filled square with text, so this should be fine. If `32Ã—32.png` looks muddy, hand-tweak that specific render or commit a 32Ã—32-optimized source.
- **Tray icon contrast in dark mode** â€” if the template-tinted "M" reads too thin against a dark menu bar, bump the silhouette's stroke weight. Easy to iterate; just re-export the PNGs.
- **`Image::from_path` resource resolution differs between `tauri dev` and bundled `.app`.** The bundled path resolves under `Contents/Resources/`; dev resolves under the dev cargo target. The `BaseDirectory::Resource` path tag handles both, but worth a smoke test before claiming v1 is done.

## Open Questions

1. **Hand-rolled tray SVG vs. extracting from the brand SVG.** A standalone tray.svg is cleaner and decouples brand tweaks from menu-bar appearance. Lean **standalone** but if the brand glyph already reads well as a monochrome silhouette, just isolate that path and use it directly.
2. **Should `app:icons` also regenerate the CLI's `AppIcon.icns`?** The CLI build today uses `build/macos/app-template/Contents/Resources/AppIcon.icns`; a unified regen script would keep CLI and shell in sync. Lean **yes, follow-up commit** â€” not in v1 scope to avoid bundling concerns, but the path is one rsync away.
3. **Linux tray icon.** Linux desktops vary; many don't honor "template" semantics. If/when we ship a `.deb` or `.AppImage`, use the colored bundle icon as the tray icon on Linux. Out of v1.
4. **macOS notarization** â€” the icon swap doesn't affect the entitlements file. No expected re-notarization friction; flagged for awareness.
